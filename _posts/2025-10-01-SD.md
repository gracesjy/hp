### ✅ 개요

UML 도 그러하고, 제조 쪽에서는 시퀀스 다이어그램(Sequence Diagram) 을 많이 사용한다. 아무래도 노드간 메시지들의 시간 선에서 데이터나 메시지의 흐름을 보기 쉽기 때문이 아닐까 한다. <br>
아주 오래 전 JAVA RCP 로 개발된 솔루션에 약간 관여 했었는데 그것에는 부가적으로 시퀀스다이어그램을 보여주어 각 시스템간 통신, 데이터등을 시각적으로 보여주는 기능이 있었다. <br>
문제는 JAVA RCP 의 유지보수가 참 쉽지가 않고, 그 GUI 도 사실 촌시럽고 배포도 그렇고 .. 여러가지로 애물 단지가 되어 버렸다.<br><br>
어째튼 생각이 나서 AI 에게 약간의 이미지와 데이터를 주고 한번 만들어 보라고 했더니 React 소스를 던져 주었는데 이것을 해 조금씩 수정해 보니, 이렇게 쉬웠나 ? 하는 생각이 들었다. <br><br>

적당한 개발자를 구하기 어려웠었는데, 이제는 필요가 없을 듯 하고 그들을 설명해 주는 것보다 AI 에게 설명하는 것이 일반적인 경우에는 훨씬 빠른 것 같다는 개인적인 생각이 요즘은 든다.<br>

### ✅ Viewer
이 소스는 Copilot 과 ChatGPT 의 도움을 많이 받았다.  다만 데이터 구성 등은 직접 설계 하였지만 큰 어려움은 없었으며 예전에 JAVA 로 많은 구현을 했었고 Python 과 Vue.js 관련 프로젝트를 해 보았기 때문에, React 프론트엔드, FastAPI 백엔드로 금방 구현이 가능했다.  정확하게 2시간 남짓 걸렸다.<br>
SVG 는 2개를 만드는데, 하나는 출력을 위한 것이다.  Scroll 시에 Node 가 사라지면 안되기 때문에 노드는 iframe 구성을 한 것이지만, 출력(Save)할 때는 모두 한꺼번에 나타나면 된다. <br>

### ✅ 후기

Editor 까지 만들 필요가 없었다. 데이터를 잘 적재하면 자동으로 Sequence Diagram 에 필요한 데이터가 만들어지는 구조 때문에 Viewer 만 있어도 큰 문제가 없다.  즉, 단순히 Diagram 을 그리는 업무가 아니기 때문이다. <br>

Moving 콤보를 만드는 것도 AI 는 어렵지 않게 했고, 데이터 구조에만 신경 쓴다면 이런 것들을 만드는 것은 AI 와 손쉽게 할 수 있을 듯 하다.
